# Knowledge-Doc 工作流

## 核心理念

**文档是知识的载体**：业务文档必须以实际代码为基础，真实反映代码的实现逻辑。文档跟着代码走，代码变更后必须同步更新文档。

**代码分析是主手段，GitNexus 是补充**：先通过全量阅读代码来理解业务逻辑，再使用 GitNexus 工具补充检查是否有遗漏的引用关系和影响范围。

---

## 文档目录结构

```
doc/
├── 业务文档.md              # 入口索引：所有业务模块汇总
├── 版本记录.md              # 变更历史：文档版本追踪
└── {模块}.md               # 领域文档：直接放在 doc 目录下
```

## 工作流程

### 场景一：生成新模块文档

**触发条件**：新功能开发完成，需要为其创建业务文档

```
1. 初始化文档目录
   ├── mkdir -p doc/（如果不存在）
   ├── 创建 doc/业务文档.md（如果不存在）
   └── 创建 doc/版本记录.md（如果不存在）

2. 全量阅读代码，分析业务逻辑
   ├── 阅读入口文件（如 handler.go），识别业务入口
   │   └── 理解：有哪些入口点、请求参数、路由分发逻辑
   ├── 阅读核心逻辑文件（如 controller.go, service.go）
   │   └── 理解：业务如何分发、核心逻辑是什么、参数校验
   ├── 阅读数据访问文件（如 dao.go, repository.go）
   │   └── 理解：有哪些数据操作、缓存策略
   ├── 阅读数据模型文件（如 model.go, schema.go）
   │   └── 理解：数据结构、字段含义
   └── 阅读外部依赖调用
       └── 理解：调用了哪些外部服务、数据来源

3. 梳理完整的业务流程
   ├── 确认有哪些业务操作（增删改查等）
   ├── 确认每个操作的完整调用链路
   ├── 确认涉及的数据模型和表结构
   └── 确认外部依赖和影响范围

4. GitNexus 补充检查（可选）
   ├── gitnexus_query({query: "<业务概念>"}) → 确认执行流程
   └── gitnexus_impact({target: "<核心符号>", direction: "upstream"}) → 检查是否有遗漏调用方

5. 生成领域文档
   ├── 模块概述（核心职责一句话描述）
   ├── 业务入口（所有入口点和操作）
   ├── 业务逻辑详解（每个操作的具体逻辑）
   ├── 数据模型（表结构和字段）
   └── 影响范围（依赖的服务、风险点）

6. 更新业务文档索引
   └── 在业务文档.md 添加模块索引

7. 记录版本变更
   └── 在版本记录.md 添加变更记录
```

### 场景二：更新现有模块文档

**触发条件**：代码变更导致现有文档过时

```
1. 阅读变更代码
   ├── git diff 查看变更内容
   ├── 理解新逻辑
   └── 确定影响范围

2. GitNexus 分析影响
   ├── gitnexus_detect_changes({scope: "unstaged"}) → 识别变更文件
   └── gitnexus_impact({target: "<变更符号>", direction: "downstream"}) → 分析影响范围

3. 更新领域文档
   ├── 修改对应章节
   └── 保持文档结构一致

4. 记录版本变更
   └── 在版本记录.md 添加变更记录
```

## 代码分析方法

### 代码结构梳理

**第一步：识别入口文件**
- 找到处理请求的文件（如 `*handler.go`、`*controller.go`）
- 识别有多少个业务入口
- 理解各入口的职责定位

**第二步：梳理调用关系**
- 从入口方法开始，追踪完整调用链
- 记录每个调用层级涉及的文件和方法
- 明确数据在各层之间的流转

**第三步：整理模块划分**
- 理解各模块的职责边界
- 识别模块间的依赖关系
- 确定核心逻辑所在位置

### 业务逻辑理解

**参数校验**：
- 必填参数有哪些
- 参数格式校验规则（类型、长度、正则等）
- 校验失败返回的错误类型和错误码

**核心业务逻辑**：
- 业务分发逻辑（如根据 template_type 调用不同处理）
- 业务判断条件（如权限判断、状态判断、AB测试分流）
- 数据转换过程（输入如何变成输出）

**数据操作**：
- 查询了哪些数据表
- 数据缓存策略（缓存 key、过期时间）
- 数据的组装和格式化逻辑

**外部依赖**：
- 调用了哪些外部服务/RPC
- 外部服务的调用时机和作用
- 外部服务返回数据的处理方式

**返回输出**
- 返回数据的数据结构

### 阅读理解要求

**理解代码入口**：
- 找到处理请求的文件（通常是 `*handler.go`）
- 识别有哪些公开方法
- 理解每个方法的请求参数和返回值类型
- 理解参数从哪里获取（URL参数、请求体、Header、Cookie等）

**理解调用链路**：
- 追踪从入口到数据层的完整调用链
- 记录每个步骤涉及的文件：函数
- 理解数据在各步骤间的转换
- 返回数据的数据结构

---

### 文档结构抽象原则

**禁止硬编码分层名称**：
- ❌ 不要写"Handler层"、"Service层"、"DAO层"
- ❌ 不要写"Handler负责转发"
- ✅ 用业务语言描述："登录入口"、"商品查询"、"订单创建"

**使用业务语言 + 文件引用**：
- ✅ "创建落地页：pkg/admin/handler/dynamic_landing_page.go:DynamicLandingPagesHandler.Create"
- ✅ "用户通过首页入口访问落地页，pkg/web/handler/dynamic_landing_page.go:LandingPagesHandler.Get"
- ✅ "管理员可以通过后台创建落地页，需要提供标题、内容链接、模板类型"

## 领域文档格式指南

### 文档头部

```markdown
# {模块名称}

## 1. 概述

{模块功能的一句话说描述}

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| {职责1} | {说明} |
| {职责2} | {说明} |

### 1.2 适用场景

- {场景1}
- {场景2}
```

### 业务入口

```markdown
## 2. 业务入口

### 2.1 {入口名称1}

**入口文件**：src/xxx/handler/xxx.go:{Handler名}

**定位**：{描述}

| 操作 | 说明 | 关键参数 | 入口方法 |
|------|------|----------|----------|
| {操作1} | {说明} | {param1}, {param2} | {file}:{Handler}.{Method} |

**{操作名称}**

- 文件：`pkg/xxx/handler/xxx.go:{Handler}.{Method}`
- 说明：{描述}

**请求示例**：
```
{method} {path}
{
  "field1": "value1"
}
```

**响应示例**：
```
{
  "code": 0,
  "data": {
    "field1": "value1"
  }
}
```

### 2.2 {入口名称2}

{描述}
```

### 业务逻辑详解

```markdown
## 3. 业务逻辑详解

### 3.1 {操作名称}

**操作说明**：{描述}

**入口文件**：`{file}:{Struct}.{Method}`

**输入**：
- {参数1}：{类型}，{说明}
- {参数2}：{类型}，{说明}

**处理流程**：
```
1. 参数校验
   ├── 文件：{校验文件}:{函数}
   └── 如果 {condition}，返回错误 "xxx"

2. 数据查询
   ├── 文件：{查询文件}:{函数}
   └── 缓存 key: {key}

3. 业务处理
   ├── 步骤1：{action1} → {file}:{函数}
   ├── 步骤2：{action2} → {file}:{函数}
   └── 步骤3：{action3} → {file}:{函数}

4. 返回结果
```

**输出**：
- {返回字段1}：{类型}，{说明}
- {返回字段2}：{类型}，{说明}

**涉及的表**：
- `{table1}` - {说明}
- `{table2}` - {说明}

**外部依赖**：
- {服务1} - {说明}
- {服务2} - {说明}
```

### 数据模型

```markdown
## 4. 数据模型

### 4.1 {表名1}

| 字段 | 类型 | 说明 |
|------|------|------|
| {field1} | {type} | {说明} |
| {field2} | {type} | {说明} |

**关联关系**：
- 与 `{table2}` 一对多，通过 `{field}` 关联

### 4.2 {表名2}

{说明}
```

### 影响范围

```markdown
## 5. 影响范围

### 5.1 外部依赖

| 服务 | 依赖方式 | 说明 |
|------|----------|------|
| {服务1} | RPC 调用 | {说明} |
| {服务2} | RPC 调用 | {说明} |

### 5.2 风险评估

| 操作 | 风险等级 | 影响说明 |
|------|----------|----------|
| {操作1} | HIGH | {说明} |
| {操作2} | MEDIUM | {说明} |
```

## 注意事项

- **代码优先**：先全量阅读代码理解业务逻辑，再写文档
- **业务语言**：用业务术语描述，同时标注文件路径和函数名
- **流程完整**：每个操作都要有完整的输入→处理→输出描述
- **结构统一**：同类文档保持相同格式
- **保持同步**：代码变更后立即更新文档
