# Spec-Dev 工作流

## 核心理念

**规范先行 (Spec-First Development)**：
1. 先理解业务文档，明确需求
2. 使用 openspec 工具创建和管理 SPEC 规范
3. 基于 SPEC 实现代码
4. 实现完成后更新业务文档

## 工作流程

### Step 1: 需求收集

当用户触发 spec-dev 时：

1. **询问模式**：
```
📋 Spec-Dev 工作流

请选择操作模式：
1. 🆕 新建规范 - 开始一个新的功能开发
2. 🔄 迭代更新 - 在开发中发现问题，更新现有规范

请输入选项或直接描述你的需求：
```

2. **根据用户选择**：
   - **选择"新建"**：继续下面的需求来源收集
   - **选择"迭代更新"**：询问现有变更名称，读取现有规范，然后进入迭代流程

3. **读取相关业务文档**：
   - 读取 `/doc/业务文档.md` 作为入口索引
   - 根据需求内容读取对应的功能文档
   - 整理需求要点

### Step 2: 使用 openspec 创建变更

**使用 openspec 工具管理规范文件**：

```bash
# 1. 创建新变更（会自动创建目录结构）
openspec new change {变更名称}

# 2. 查看变更状态
openspec status --change {变更名称}

# 3. 获取下一步指导
openspec instructions {artifact-id} --change {变更名称}
```

**变更目录结构**（由 openspec 自动创建）：
```
openspec/changes/{变更名称}/
├── .openspec.yaml      # 元数据配置
├── proposal.md         # 需求提案
├── design.md           # 设计决策
├── tasks.md           # 任务拆分
└── specs/
    └── {功能名称}/
        └── spec.md     # 详细规格说明
```

### Step 3: 按顺序创建规范文件

使用 `openspec instructions` 获取每个 artifact 的创建指导：

1. **proposal.md** (需求提案)
   - 背景与目标
   - 非目标（明确不包括的范围）
   - 用户故事

2. **design.md** (设计决策)
   - 方案选择与理由
   - 详细设计（数据结构、接口）
   - 影响范围与风险

3. **tasks.md** (任务拆分)
   - 按模块/阶段拆分任务
   - 任务间依赖关系

4. **specs/{功能}/spec.md** (详细规格)
   - 使用 Gherkin 风格语法
   - Requirement + Scenario 结构

### Step 4: 用户确认规范

创建完所有规范文件后：
- 展示给用户确认
- 用户提出修改意见
- 迭代直到用户确认

### Step 5: 实现代码

用户确认规范后：
- 使用 `openspec apply {变更名称}` 开始实现
- 或手动基于 tasks.md 实现
- 每个实现步骤前引用规范中的对应需求
- 遇到问题时记录到规范注释中

### Step 6: 更新业务文档

实现完成后：
- 使用 `knowledge-doc` skill 更新目录下的相关业务文档
- 如果业务文档有变化，帮用户更新
- 记录变更历史

### Step 7: 开发中发现问题时的迭代

**适用场景**：开发过程中发现实现问题或业务问题，需要回溯更新规范

```
┌─────────────────────────────────────────────────────────────┐
│  开发中发现问题 → 迭代更新规范                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  开发中遇到：                                                │
│  ├── 设计与实现冲突，需要调整设计                          │
│  ├── 发现遗漏的需求场景                                    │
│  ├── 发现现有模块也需要配合修改                            │
│  └── 业务逻辑理解有误，需要修正                           │
│                                                             │
│  处理流程：                                                  │
│  1. 记录问题（代码注释或 TODO）                          │
│  2. 评估影响范围（规范/业务文档/其他代码）                │
│  3. 更新 openspec 规范（proposal/design/tasks）           │
│  4. 更新业务文档                                           │
│  5. 继续实现                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**问题分类处理**：

| 问题类型 | 处理方式 | 需要更新的文档 |
|---------|---------|---------------|
| 设计遗漏 | 更新 design.md + tasks.md | 规范 + 业务文档 |
| 需求变更 | 更新 proposal.md + design.md | 规范 + 业务文档 |
| 技术方案调整 | 更新 design.md | 规范 |
| 现有模块联动 | 更新 tasks.md + 业务文档 | 规范 + 业务文档 |

**关键原则**：

```
1. 不要跳过规范直接改代码
2. 规范是代码的"宪法"，修改代码前先修改规范
3. 发现问题时，先问："规范应该怎么改？"
4. 更新规范后，再评估对业务文档的影响
```

## openspec 常用命令

| 命令 | 说明 |
|------|------|
| `openspec list` | 列出所有变更 |
| `openspec new change {name}` | 创建新变更 |
| `openspec status --change {name}` | 查看变更状态 |
| `openspec instructions {artifact} --change {name}` | 获取 artifact 创建指导 |
| `openspec apply {name}` | 开始实现任务 |
| `openspec verify {name}` | 验证实现完整性 |
| `openspec archive {name}` | 归档完成的变更 |

```
- [ ] 如有新功能，添加到业务文档索引
- [ ] 归档变更：`openspec archive {变更名称}`

是否需要我帮你更新业务文档或归档变更？
```

## 注意事项

- **使用 openspec 工具**: 优先使用 openspec 命令管理变更，而非手动创建文件
- **Spec First**: 永远先创建规范再实现，不要边写边改
- **引用规范**: 实现时引用规范中的 Requirement ID，便于追溯
- **文档同步**: 实现完成后务必更新业务文档，保持文档新鲜度
- **简洁清晰**: 规范描述要清晰，避免模糊表述
