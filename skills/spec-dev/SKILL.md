# Spec-Dev 工作流

## 核心理念

**规范先行 (Spec-First Development)**：
1. 先理解业务文档，明确需求
2. 使用 openspec 工具创建和管理 SPEC 规范
3. 基于 SPEC 实现代码
4. 实现完成后更新业务文档

## 工作流程

### Step 1: 需求收集

> 🔴 **上下文中被压缩时，认准此步** — 重新读取 SKILL.md 确认流程

当用户触发 spec-dev 时：

1. **询问模式**：
```
📋 Spec-Dev 工作流

请选择操作模式：
1. 🆕 新建规范 - 开始一个新的功能开发
2. 🔄 迭代更新 - 在开发中发现问题，更新现有规范

```

2. **根据用户选择**：

3. **读取相关业务文档**：
   - 读取 `/doc/业务文档.md` 作为入口索引
   - 根据需求内容读取对应的功能文档
   - 整理需求要点

### Step 2: 使用 openspec 创建变更

> 🔴 **上下文中被压缩时，认准此步** — openspec 命令在此阶段使用

**使用 openspec 工具管理规范文件**：

```bash
# 1. 创建新变更（会自动创建目录结构）
openspec new change {变更名称}

# 2. 查看变更状态
openspec status --change {变更名称}

# 3. 获取下一步指导
openspec instructions {artifact-id} --change {变更名称}
```

**变更目录结构**（由 openspec 自动创建）：
```
openspec/changes/{变更名称}/
├── .openspec.yaml      # 元数据配置
├── proposal.md         # 需求提案
├── design.md           # 设计决策
├── tasks.md           # 任务拆分
└── specs/
    └── {功能名称}/
        └── spec.md     # 详细规格说明
```

> 🔴 **完成本步后，必须输出 `[STEP 2 DONE]` 再继续**
> 🔴 **如果上下文超长，重新 view 本文件确认流程**

### Step 3: 按顺序创建规范文件

> 🔴 **上下文中被压缩时，认准此步** — 按 proposal → design → tasks → spec.md 顺序创建

使用 `openspec instructions` 获取每个 artifact 的创建指导：

1. **proposal.md** (需求提案)
   - 背景与目标
   - 非目标（明确不包括的范围）
   - 用户故事

2. **design.md** (设计决策)
   - 方案选择与理由
   - 详细设计（数据结构、接口）
   - 影响范围与风险

3. **tasks.md** (任务拆分)
   - 按模块/阶段拆分任务
   - 任务间依赖关系

4. **specs/{功能}/spec.md** (详细规格)
   - 使用 Gherkin 风格语法
   - Requirement + Scenario 结构

> 🔴 **上下文中被压缩时，认准此步** — 用户确认后进入实现阶段

### Step 7: 迭代更新流程

> 🔴 **上下文中被压缩时，认准此步** — 用于更新现有规范，非新建

**适用场景**：用户主动要求更新迭代现有规范，或开发过程中发现问题需要回溯更新规范

**⚠️ 重要前提**：更新迭代时，用户必须主动提供 spec（变更名称或文件路径），无法自行查找

#### 7.1 触发迭代时的用户交互

```
🔄 规范迭代更新

请提供需要更新的 spec：

1. 变更名称（openspec/changes/{变更名称}）
   └── 或直接提供 .openspec.yaml 文件路径

2. 需要更新的部分（可多选）：
   ├── □ proposal.md - 需求提案变更
   ├── □ design.md - 设计决策变更
   ├── □ tasks.md - 任务拆分变更
   └── □ specs/{功能}/spec.md - 详细规格变更

3. 简要描述本次变更内容：
   └── {请描述变更的核心内容}

请提供以上信息，我将帮你完成规范迭代。
```

#### 7.2 迭代更新流程

```
┌─────────────────────────────────────────────────────────────┐
│  迭代更新流程                                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 【用户提供】确认变更目标                                  │
│     └── 用户提供变更名称和变更内容描述                        │
│                                                             │
│  2. 【协助分析】评估影响范围                                  │
│     ├── 读取现有规范文件                                     │
│     ├── 分析变更对现有规范的影响                             │
│     └── 识别需要同步更新的相关文档                           │
│                                                             │
│  3. 【更新规范】按需更新规范文件                              │
│     ├── proposal.md（需求变更时）                           │
│     ├── design.md（设计变更时）                             │
│     ├── tasks.md（任务变更时）                               │
│     └── specs/{功能}/spec.md（规格变更时）                  │
│                                                             │
│  4. 【确认规范】用户确认更新后的规范                          │
│                                                             │
│  5. 【实现代码】按更新后的规范实现代码                        │
│     ├── 基于更新后的 tasks.md 执行实现                       │
│     ├── 实现时引用规范中的 Requirement ID                   │
│     └── 遇到问题记录到规范注释中                             │
│                                                             │
│  6. 【更新业务文档】使用 knowledge-doc skill                 │
│     └── ⚠️ 必须调用 knowledge-doc skill 更新业务文档        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

> 🔴 **完成本步后，必须输出 `[STEP 7 DONE]` 再继续**
> 🔴 **如果上下文超长，重新 view 本文件确认流程**

#### 7.3 问题分类处理

| 问题类型 | 处理方式 | 需要更新的文档 |
|---------|---------|---------------|
| 设计遗漏 | 更新 design.md + tasks.md | 规范 + 业务文档（必须用 knowledge-doc） |
| 需求变更 | 更新 proposal.md + design.md | 规范 + 业务文档（必须用 knowledge-doc） |
| 技术方案调整 | 更新 design.md | 规范 |
| 现有模块联动 | 更新 tasks.md | 规范 + 业务文档（必须用 knowledge-doc） |

#### 7.4 关键原则

```
1. 不要跳过规范直接改代码
2. 规范是代码的"宪法"，修改代码前先修改规范
3. 发现问题时，先问："规范应该怎么改？"
4. 更新规范后，必须回到代码实现（流程第5步）
5. ⚠️ 业务文档更新必须使用 knowledge-doc skill
```

## openspec 常用命令

| 命令 | 说明 |
|------|------|
| `openspec list` | 列出所有变更 |
| `openspec new change {name}` | 创建新变更 |
| `openspec status --change {name}` | 查看变更状态 |
| `openspec instructions {artifact} --change {name}` | 获取 artifact 创建指导 |
| `openspec apply {name}` | 开始实现任务 |
| `openspec verify {name}` | 验证实现完整性 |
| `openspec archive {name}` | 归档完成的变更 |

## 注意事项

- **使用 openspec 工具**: 优先使用 openspec 命令管理变更，而非手动创建文件
- **Spec First**: 永远先创建规范再实现，不要边写边改
- **引用规范**: 实现时引用规范中的 Requirement ID，便于追溯
- **文档同步**: 实现完成后务必更新业务文档，保持文档新鲜度
- **⚠️ 业务文档更新必须使用 knowledge-doc skill**
- **简洁清晰**: 规范描述要清晰，避免模糊表述
- **步骤锚点**: 每步完成后输出 `[STEP X DONE]`，上下文压缩时可快速定位进度
- **上下文保护**: 如果上下文超长，重新 view 本文件确认当前步骤
